# daily-news-workflow 重构计划

> 目标：清理项目中的冲突、重复和不一致，建立清晰的架构分层

---

## 一、发现的核心问题

### 1. Skill 层重复与注册混乱

**问题现象**：
```
.claude/skills/
├── daily-news/SKILL.md      # 描述 /daily-news 命令和 PDF 输出
└── news-collector/SKILL.md  # 描述新闻收集功能
```

**冲突点**：
| 冲突 | daily-news skill | news-collector skill |
|------|------------------|----------------------|
| 注册状态 | 未注册（settings.json 中不存在） | 已注册 |
| 声称的命令 | `/daily-news` | 无 |
| 声称的输出 | PDF | JSON + HTML/Markdown/RSS |
| 实际代码 | 不存在 `/daily-news` 命令 | `fetch-news.py` 实际存在 |

**根因**：`daily-news` skill 是早期设计文档，实际开发时转向了不同的实现方向，但未清理旧文档。

---

### 2. Agent 定义与实际代码严重不符

**Agent 文档描述** vs **实际代码实现**：

| 组件 | Agent 文档声称 | 实际代码 | 差距 |
|------|---------------|----------|------|
| **输出格式** | `tech-news-digest-YYYY-MM-DD.pdf` | `docs/index.html` | 完全不符 |
| **RSS 源数量** | 8 个（4 英文 + 4 中文） | 3 个（硬编码） | 配置未读取 |
| **PDF 生成** | 使用 Playwright | 未使用 | 依赖不存在 |
| **架构流程** | 4 Stage（含 Fan-out 并行） | 顺序 6 阶段 | 描述不符 |

**根因**：Agent 文档先于代码编写，代码实现时做了务实调整（如放弃 PDF 改用 HTML），但未同步更新文档。

---

### 3. Commands 文档与可执行性脱节

**致命误解**：

```json
// .claude/settings.json 中的注册
{
  "commands": {
    "fetch-news": { "file": ".claude/commands/fetch-news.md" }
  }
}
```

**用户实际体验**：
```bash
cd /path/to/daily-news-workflow
claude
# 在 Claude Code 中输入：
/fetch-news
# 结果：❌ 命令不存在
```

**核心问题**：
- `.claude/settings.json` **不会**被 Claude Code 自动加载
- `.claude/commands/*.md` 只是**设计文档**，不是可执行命令
- 文档中的 Stage 结构（Stage 1/2/3）与代码实际阶段（Stage 1/2/2b/2c/3/4）不一致

---

### 4. 配置文件与代码硬编码脱节

**配置文件中定义**（`config/news-sources.json`）：
```json
{
  "rss": [
    {"name": "TechCrunch", "url": "...", "priority": "high"},
    {"name": "The Verge", "url": "...", "priority": "high"},
    {"name": "36氪", "url": "...", "priority": "high"},
    {"name": "极客公园", "url": "..."}  // 代码未使用
  ]
}
```

**代码中硬编码**（`scripts/fetch-news.py`）：
```python
SOURCES = [
    {"name": "TechCrunch", "url": "https://techcrunch.com/feed/"},
    {"name": "The Verge", "url": "https://www.theverge.com/rss/index.xml"},
    {"name": "36氪", "url": "https://36kr.com/feed"},
]
```

**后果**：修改 `config/news-sources.json` 不会影响程序行为，配置失去意义。

---

### 5. 输出目录定义混乱

| 位置 | 声称的输出目录 | 实际输出目录 |
|------|---------------|--------------|
| `fetch-news.md` command | `cache/daily-news.json` | ✅ 一致 |
| `config/news-sources.json` | `./output` | ❌ 未使用 |
| `generate-all.py` | `docs/` (HTML), `docs/` (Markdown) | ✅ 部分一致 |
| `daily-news` skill | `./output/tech-news-digest-YYYY-MM-DD.pdf` | ❌ 未生成 |

### 6. 架构理解错误（最根本问题）

**误解链条**：

```
用户认知                    实际情况
─────────────────────────────────────────────────────────────
.claude/settings.json       ≠  自动注册命令
    ↓                           ↓

    ↓                           ↓
Claude Code 能直接执行      ≠  实际靠 GitHub Actions 执行
    ↓                           ↓
这是一个可被 Claude Code    =  正确，但作用是供 WorkflowProgram-CN
直接调用的工作流               的 /evolve-workflow 审计和演进
```

## 二、解决方案

### 方案概述

采用**分层清理策略**：

1. **保留层**：核心 Python 代码（功能已验证）
2. **修复层**：配置与代码的绑定
3. **清理层**：删除误导性文档
4. **明确层**：README 清晰说明架构定位

---

### 阶段 1：清理误导性文档（优先级：高）

#### 1.1 删除重复的 skill

**操作**：删除 `.claude/skills/daily-news/` 目录

**理由**：
- 未在 `settings.json` 中注册
- 描述的 `/daily-news` 命令不存在
- 描述的 PDF 输出不存在
- `news-collector` 已足够覆盖需求

**保留**：`.claude/skills/news-collector/SKILL.md`（已注册且描述与实际最接近）

#### 1.2 更新 news-collector skill 以匹配实际

**修改内容**：
```yaml
---
name: news-collector
description: 新闻收集核心技能，支持多源聚合、内容筛选和翻译
version: 1.1.0  # 升级版本号
---

# News Collector Skill

## 能力范围

1. **多源收集**: RSS、NewsAPI、GitHub Trending、HuggingFace
2. **内容筛选**: 基于关键实体的相关性评分
3. **自动翻译**: 英文 → 中文（使用 deep_translator）
4. **多格式输出**: HTML、Markdown、RSS（非 PDF）

## 执行方式

### GitHub Actions（推荐）
- 每天 UTC 01:00 自动执行
- 输出部署到 GitHub Pages

### 本地运行
```bash
# 收集新闻
python scripts/fetch-news.py --output cache/daily-news.json

# 生成多格式输出
python scripts/generate-all.py --input cache/daily-news.json --output-dir docs --summary
```

## 配置

编辑 `config/news-sources.json` 自定义：
- RSS 源列表
- API 参数
- 筛选实体和评分权重
- 输出设置
```

#### 1.3 删除或重构 commands 目录

**方案 A（推荐）：删除所有 commands**

理由：这些文档会误导用户认为可以直接在 Claude Code 中 `/fetch-news`，实际上：
- 用户需要在终端运行 `python scripts/fetch-news.py`
- 或在 GitHub Actions 中自动执行

**方案 B（备选）：保留为设计文档，但重命名**

将 `.claude/commands/*.md` 移动到 `.claude/design/` 或添加显著说明：
```markdown
> ⚠️ **注意**：本文档是设计规格说明，**不是可执行命令**。
> 实际执行请运行：`python scripts/fetch-news.py`
```

**推荐采用方案 A**，因为：
- 设计文档与实际代码不符（Stage 命名、PDF 输出等）
- 保留会 perpetuate 误解
- WorkflowProgram-CN 演进时会重新生成正确的设计文档

### 阶段 2：修复配置与代码的绑定（优先级：高）

#### 2.1 修改 `scripts/fetch-news.py` 读取配置文件

**当前代码**（硬编码）：
```python
class RSSCollector:
    SOURCES = [
        {"name": "TechCrunch", "url": "https://techcrunch.com/feed/"},
        {"name": "The Verge", "url": "https://www.theverge.com/rss/index.xml"},
        {"name": "36氪", "url": "https://36kr.com/feed"},
    ]
```

**修改为**（读取配置）：
```python
import json
from pathlib import Path

def load_config():
    """从 config/news-sources.json 加载配置"""
    config_path = Path(__file__).parent.parent / "config" / "news-sources.json"
    with open(config_path, "r", encoding="utf-8") as f:
        return json.load(f)

class RSSCollector:
    def __init__(self, config=None):
        self.config = config or load_config()
        self.sources = self.config["sources"]["rss"]
        # 只启用 priority=high 的源
        self.active_sources = [s for s in self.sources if s.get("priority") == "high"]
```

**同样修改**：
- `ContentFilter.KEY_ENTITIES` → 从配置读取
- `ContentFilter` 的评分权重 → 从配置读取

#### 2.2 统一输出目录定义

**修改 `config/news-sources.json`**：
```json
{
  "output": {
    "cache_dir": "./cache",        // 中间数据
    "docs_dir": "./docs",          // 最终输出（GitHub Pages 源）
    "temp_dir": "./tmp"            // 临时文件
  }
}
```

**修改 `scripts/generate-all.py`**：
```python
def main():
    parser.add_argument("--output-dir", default="./docs", help="输出目录（默认：docs/）")
```

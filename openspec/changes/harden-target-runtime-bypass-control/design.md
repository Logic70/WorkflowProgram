# Design: Target Runtime Bypass Hardening

## Decision Frame

Goal: prevent generated target workflows from bypassing the managed runtime after a runtime failure.

Scope: target-side runtime generated by WorkflowProgram, command wrapper validation, finalizer-owned publish state, and deterministic regression tests. WorkflowProgram product-level `runtime_host.py` may still use `claude_cli`; this change only governs generated target workflow executors and publish finalization.

Compatibility truth: v0.1.14 target runtime already supports `fixture_host`, `command_adapter`, `current_agent`, and `manual`, rejects target `claude_cli`, writes run-scoped `target-state.json`, `node-results.json`, `artifact-provenance.json`, and publishes only through `target-runtime-finalizer.py`.

## Runtime Ownership

For `target_runtime_policy.mode=managed_runtime`, the public command is not an orchestration document. It is a thin wrapper whose only job is to invoke `.workflowprogram/runtime/workflow-entry.py` and explain where to find runtime evidence. It must not contain node-by-node execution instructions, direct report assembly instructions, final output copy steps, doctor invocations, or manifest/latest marker writing instructions.

Rich guidance belongs in:

- `workflow_graph.nodes[*]`
- target design source documents
- node design documents
- owner skill/agent prompts
- required reports consumed by the finalizer

`target_publish_policy.required_reports` must only list gate reports that already exist in the current `RUN_ROOT` before finalizer execution. It must not list finalizer-owned files such as `manifest_path` or `latest_marker`.

`workflow_graph.nodes[*].input_refs` and `workflow_graph.nodes[*].output_refs` must also not reference finalizer-owned manifest/latest-marker paths. Business nodes cannot read or write the final publish transaction state before the finalizer runs.

## Publish State Doctor

`validate-target-publish-state.py` validates the final publish directory after finalizer publish or during S5/audit checks. It consumes `workflow-spec.yaml`, `TARGET_ROOT`, and optionally `RUN_ROOT`.

It fails when:

- final manifest is missing or not finalizer-owned;
- manifest status/verdict are not `COMPLETE/PASS`;
- latest marker does not reference the same run and manifest;
- run root is missing or mismatched;
- `target-state.json` is missing, not `PASS`, not `finalizer_status=PASS`, or not marked `published=true`;
- `node-results.json` contains failed nodes;
- `artifact-provenance.json` is missing or lacks artifacts;
- any required report fails its declared `status_field/pass_values`;
- published artifacts listed in manifest are missing or have digest mismatches.

## Finalizer Seal

The finalizer writes manifest fields:

- `producer: target-runtime-finalizer.py`
- `finalizer_status: PASS`
- `source_state_status`
- `state_sha256`
- `node_results_sha256`
- `artifact_provenance_sha256`
- `required_reports`

The seal is not meant to be cryptographic tamper-proof. It is a machine-checkable ownership and consistency marker so later validators can reject hand-written `COMPLETE` manifests.

## Current-Agent And Manual Execution

`current_agent` and `manual` still use evidence-first semantics:

1. The operator/model writes run-scoped node outputs under `RUN_ROOT`.
2. The operator/model writes `outputs/stages/executor-evidence/<node-id>.json`.
3. The runner validates evidence and produces `BLOCKED`, not `PASS`.
4. The finalizer revalidates evidence, reports, provenance, and publishes atomically.

If runtime is started before evidence exists, it must fail or block without publishing. The command must not instruct the model to repair this by copying outputs to final publish directories.

## Review Closure

Round 1 accepted finding: command wrapper validation cannot live only in generated runtime validation because migrated workflows may be edited after generation. Fix: add standalone publish-state doctor and extend generated runtime validator.

Round 2 accepted finding: finalizer manifest hashes could become circular if computed after `target-state.json` mutation. Fix: hash the pre-publish state and record it as `state_sha256`; doctor validates live state fields rather than requiring post-publish state hash equality.

Round 3 accepted finding: product-level `claude_cli` should not be conflated with target executor providers. Fix: design explicitly scopes the ban to generated target runtime executors.

Round 4 accepted finding: required reports can create a circular dependency if they include the finalizer-owned manifest/latest marker. Fix: spec validation rejects those paths in `required_reports`.

Round 5 accepted finding: workflow graph nodes can reintroduce stale-output dependency by reading/writing finalizer-owned manifest/latest marker paths. Fix: spec validation rejects those paths in node `input_refs` and `output_refs`.

Round 6 found no new actionable issue.

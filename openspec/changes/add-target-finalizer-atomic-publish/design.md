# Design: Target Runtime Finalizer Atomic Publish

## Decision Frame

- Goal: generated target workflows must not produce contradictory final state, latest marker, manifest, and report evidence.
- Scope: target-side runtime generated by WorkflowProgram and shared scripts it delegates to.
- Existing truth: `target_runtime_policy.mode=managed_runtime` already makes `target-workflow-runner.py` own `workflow_graph.nodes` execution and provenance.
- New artifact: `target_publish_policy` owns final publish semantics.

## Design

`target_publish_policy.enabled=true` changes target node output materialization from `TARGET_ROOT/<output_ref>` to `RUN_ROOT/<output_ref>`. Nodes still declare normal domain paths such as `outputs/target-workflow/report.html`, but the runner writes the current run copy first.

After `validate-target-runtime-state.py` passes, `workflow-entry.py` invokes `target-runtime-finalizer.py`. The finalizer loads only current-run evidence:

- `target-state.json`
- `node-results.json`
- `artifact-provenance.json`
- `target_publish_policy.required_reports`

If all required evidence is current and passing, finalizer stages files under a temporary publish directory, writes current-run `run-manifest.json` and latest marker, then atomically replaces `TARGET_ROOT/<publish_root>`. If any check fails, finalizer patches `target-state.json` to `FAIL/implementation` and does not publish.

## Review Closure

Round 1 accepted issue: run-scoped outputs would break downstream node inputs if inputs only searched `TARGET_ROOT`. Fix: runner searches both output root and target root for inputs.

Round 2 accepted issue: old workflows could regress if finalizer is globally required. Fix: policy is optional; new templates enable it, legacy specs continue unless they opt in.

Round 3 accepted issue: finalizer could overwrite control-plane paths. Fix: validator rejects publish roots under `.claude/**`, `.workflowprogram/**`, and `config/scripts/**`.

Latest review found no new actionable design issue.

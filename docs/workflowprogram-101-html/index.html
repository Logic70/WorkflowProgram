写给每一个用 Claude Code 做 workflow 的人

能跑起来只是起点，能持续跑下去才是产品。

很多 workflow 在最初几天表现很好。然后你会发现：文件被覆盖、步骤被跳过、失败无法复盘、同样的坑反复踩。 这些不是"用法不对"，而是缺少一套系统性的治理结构。WorkflowProgram 就是为了解这类问题而存在的。

这份教程会从最常见的问题出发，一步步带你理解 WorkflowProgram 的每一层设计为什么存在、怎么协作、以及你能从中借鉴什么。

第一步：看见问题

工作流编排最常见的 8 个失控点

如果你做过 Claude Code workflow，下面这些场景你大概率遇到过至少三个。 WorkflowProgram 不是凭空发明概念，它的每一层设计都是为了回应某个具体的失控场景。

1. 没有统一真源

语义散在 SKILL.md、聊天记录和人工约定里。改了文档不等于改了行为，改了行为不等于更新了文档。

回应：把执行语义收口到一份机器可读的 workflow-spec.yaml。文档只能从它派生，不能反向改语义。

2. 编排顺序靠模型记忆

这一轮先做设计再写文件，下一轮可能直接跳到写文件。步骤漏掉、重复或者乱序，全靠运气。

回应：用 workflow-entry.py 固定脚本链，用 workflow-runner.py 固定状态流转。顺序不再是"建议"，而是程序。

3. 目标项目被直接写坏

模型一旦直接覆盖目标文件，用户手工维护的内容可能丢失，冲突和回退都变得很难处理。

回应：先把候选资产写到隔离目录，再做 managed apply。冲突时保留证据，不静默覆盖。

4. 失败后无法分层定位

最后只知道"失败了"，但不知道问题出在设计层、执行层、环境层还是判定层。

回应：把 spec / runner / judge / smoke 拆成独立层，每层有自己的证据和结论。

5. 运行证据不足

只有最终产物，没有状态快照、事件流、运行 transcript。出了问题只能猜。

回应：在 RUN_ROOT 固化 context.json、state.json、events.jsonl、transcript.md、validation-runtime-report.md。

6. 验证只看退出码

"脚本跑完了"不等于"按约束执行成功了"。退出码 0 只能说明没有 crash，不能说明满足了业务约束。

回应：用 runtime_contract + test_contract + S5 judge 给出有依据的 verdict。

7. 经验不会回流

这次踩的坑、总结的教训留在聊天记录里，下一轮还是从零开始。

回应：用 S6 生成 lessons 增量，定期提炼为长期约束规则。

8. 自然语言入口漂移

同一句话这次落到 develop，下次落到 validate。入口不稳定，后面的行为就不可预期。

回应：用 workflowprogram-orchestrate + route-intent.py 做确定性路由。

第二步：理解设计哲学

Workflow 不是一段大 prompt，而是四层职责

很多 workflow 一开始就是"写一个 SKILL.md，再补几个 agent，最后改一下 settings.json"。 这种做法能快速启动，但很难持续维护。原因是它把四件本质不同的事情混在了一起。

01

真源

workflow-spec.yaml 是机器可读的执行语义真源。 从它派生出 workflow-view.md（人类可读视图）和 workflow-lowlevel.md（维护指南）， 但派生物不能反向修改语义。想改行为？改 spec。

02

控制面

workflow-entry.py 把脚本链串成固定顺序。 workflow-runner.py 负责状态转移、边界检查和证据落盘。 AI 在每个节点内理解和生成，但节点之间的跳转由程序决定。

03

验证层

生成者不能同时做裁判。workflowprogram-validate 和 workflow-s5-judge.py 消费 test_contract，独立给出 verdict。生成成功不等于验证通过。

04

闭环层

S6 把本轮的失败经验和约束候选写进 lessons.md， 稳定经验定期提炼为 constraints.md 中的 ALWAYS/NEVER 规则。 workflow 因此拥有长期记忆。

为什么是四层而不是更多？ 因为"能跑 → 可维护"之间恰好隔着这四件事：有真源、有控制面、有独立验证、有经验闭环。少一层就不完整，多一层就过度设计。

第三步：划清边界

为什么必须区分 PLUGIN_ROOT、TARGET_ROOT、RUN_ROOT

如果输入、输出、证据全混在一个目录里，你很快就会分不清"谁是参考、谁是交付、谁是中间态"。 三根目录是 WorkflowProgram 最基础的边界契约。

PLUGIN_ROOT

WorkflowProgram 自身的能力仓：skills、agents、脚本、模板。它只被读取，永远不被修改。

实际路径：dist/plugin/

TARGET_ROOT

用户的目标项目。最终交付的 .claude/ 资产在这里。只能通过 managed apply 写入，不能直接覆盖。

实际路径：用户项目根目录

RUN_ROOT

单次运行的隔离空间。spec、候选资产、状态、事件、报告全在这里，运行结束后可完整回溯。

实际路径：TARGET_ROOT/.workflowprogram/runs/<run-id>/

第四步：理解生命周期

为什么是 S0 到 S6

阶段不是为了多几个数字好看，而是为了让每一段职责都能被独立验证、回退和复盘。 设计和执行混在一起，最后出了问题只能从头来过；拆开之后，你可以精确定位到"在哪一步、因为什么"出了问题。

S0路由与目标准备
S1需求澄清
S2上下文研究
S3设计与审批
S4受控写入
S5验证判定
S6经验回流

不同意图不需要走完所有阶段。四种核心意图各有自己的阶段路径：

developS1 → S2 → S3 → S4 → S5 → S6

从需求到交付的完整链路

auditS5 → S6

对已有 workflow 做结构审计

validateS5

只做一次验证判定

iterateS6

从 lessons 中提炼改进

关键判断： 生成成功不等于验证通过（S4 ok ≠ S5 ok）；验证失败不等于这次运行没价值（S5 fail 但 S6 可以回流 lessons）。

第五步：看懂运行机制

AI 负责理解和生成，Python 负责控制面

WorkflowProgram 做了一个关键判断：需要稳定执行、稳定落盘、稳定复验的部分，不能留在 prompt 里。

所以两个角色各有分工：

  • AI：理解需求、补充设计、在每个节点内生成候选资产。
  • Python：路由意图、校验 spec、managed apply、状态转移、证据落盘、S5 判定。

第六步：分清入口关系

自然语言、orchestrate、develop、entry、runner 之间到底怎么接力

这是很多人第一次读源码时最容易混淆的地方。要点只有两个： 自然语言默认先走 orchestrate； 显式 /develop 或 workflowprogram-develop 不会再跳回 orchestrate。

路径 A：自然语言请求

  • 用户说："为当前项目设计一个 workflow"
  • workflowprogram-orchestrate 先接住请求
  • route-intent.py 判断 intent 是 develop / audit / validate / iterate
  • 再把请求分发给对应叶子入口

路径 B：显式叶子入口

  • 用户直接执行 /develop 或 workflowprogram-develop
  • 不会再回到 orchestrate skill
  • 但 workflow-entry.py 仍会再跑一次 route-intent.py
  • 目的不是重新路由 skill，而是保留路由证据、检查入口和请求是否冲突

第七步：理解写入安全

为什么不是"直接改 TARGET_ROOT/.claude/"

这是很多人第一次听到会觉得"多此一举"的设计。但只要你经历过一次"模型把你手工维护的 settings.json 覆盖了"， 你就会明白：写入安全是 workflow 工程化的硬分水岭。

直接覆盖（常见做法）

  • 用户手工维护的文件容易被误改或丢失。
  • 冲突发生时没有中间态证据，无法恢复。
  • 事后无法区分哪些文件是工具写的、哪些是人写的。

candidate → managed apply

  • AI 先把候选资产写到 RUN_ROOT/outputs/candidate/。
  • managed-assets.py plan 生成变更计划，标注每个文件的决策（create/update/conflict）。
  • managed-assets.py apply-staged 执行受控写入。冲突时保留副本，不静默覆盖。
  • managed-files.json 持续记录哪些文件是工具托管的。

第八步：分清三层验证

生成者不能同时做自己的裁判

如果"生成"和"验证"是同一个角色负责，生成成功就容易被误当成验证通过。 WorkflowProgram 把验证拆成三层，每层有明确的职责边界：

A

Runner（控制面硬约束）

检查写入边界、最小证据、状态值域、失败枚举。这些是"必须满足"的前提条件，不是"建议"。

B

S5 Judge（workflow 级判定）

消费 runtime_contract 和 test_contract，跨 entry/boundary/flow/artifacts/failure 五个维度给出 verdict。始终产出完整 check matrix，即使部分证据缺失。

C

Runtime Smoke（动态 harness）

跨 provider（fixture_host / command_adapter / claude_cli）运行端到端场景，补充动态运行态证据。它辅助主 judge，不替代主 judge。

验证产出的关键文件：

RUN_ROOT/state.json控制面状态快照
RUN_ROOT/events.jsonl事件流（可重放）
RUN_ROOT/transcript.md运行过程 transcript
validation-runtime-report.mdS5 判定报告（人类可读）
s5-validation-summary.jsonS5 判定摘要（机器可读）

第九步：理解测试

测试环节是怎么证明“工作流真的按定义和约束执行”的

WorkflowProgram 的测试不是只看命令退出码，而是分层证明： spec 合法、控制面落盘正确、S5 判定正确、端到端 smoke 能补到运行证据。

1

静态设计校验

validate-workflow.py、validate-workflow-spec.py、validate-workflow-lowlevel.py 保证仓库、spec、lowlevel 结构一致。

2

控制面校验

workflow-runner.py 和 validate-run-state.py 保证状态转移、边界、最小证据、失败类型都落在定义范围内。

3

S5 主判定

workflow-s5-judge.py 用 runtime_contract + test_contract 跨入口、边界、流程、产物、失败映射做最终 workflow verdict。

静态/半动态验证命令

python3 .claude/scripts/validate-workflow.py
python3 .claude/scripts/validate-workflow-spec.py --spec <workflow-spec.yaml>
python3 .claude/scripts/validate-workflow-lowlevel.py \
  --spec <workflow-spec.yaml> \
  --lowlevel <workflow-lowlevel.md>
python3 .claude/scripts/validate-run-state.py --state <RUN_ROOT>/state.json

运行期验证命令

python3 tools/runtime_smoke.py --fixture empty-project
python3 tools/runtime_smoke_matrix.py --json
当前验证能证明什么？ 它已经能严格证明控制面约束和最终证据是否满足定义；它不是宿主内部每一步 tool call 的全程探针，但已经不是“只看成功失败”的浅层测试。

第十步：建立长期记忆

S6 不是附录，它让 workflow 真正能进化

不会学习的 workflow 只有执行，没有演进。 S6 的核心价值是：把一次运行中积累的经验，变成下一轮运行的输入。

短期记录

lessons.md 是追加式日志。每次运行的失败经验、冲突和待提炼约束都写在这里，只追加不读取（或只读最新几条）。超过 10 条就归档。

单次运行增量

s6-lessons-delta.md 把这次运行的 run_id、failure_kind 和约束候选收成独立增量。它由 validate-lessons-delta.py 校验。

长期规则

.claude/rules/constraints.md 用 ALWAYS/NEVER 固化稳定经验。新会话只加载这个文件，不加载完整 lessons 日志，避免上下文膨胀。

WorkflowProgram 自己也在用这套模型： 维护自己的 lessons.md 和 constraints.md，用设计真源和能力矩阵避免实现漂移。它不是"只要求别人做闭环"。

第十一步：认清仓库结构

你应该把哪些目录当真源，哪些目录当派生产物

.claude/commands/源码层命令入口
.claude/skills/源码层 skills 与模板
.claude/scripts/确定性脚本链与校验器
dist/plugin/构建后的运行时载荷
tests/fixtures、expectations、transcripts
docs/HighLevel、LowLevel、教程和状态索引
维护原则： 优先修改真源文件，例如 .claude/scripts/、.claude/skills/、设计文档；dist/plugin/、PPT、报告和 view/lowlevel 都应从真源重建，不建议长期手工维护。

第十二步：落地到你的项目

想为自己的项目设计 workflow？先回答这 6 个问题

不要先写 SKILL.md。先把产品边界想清楚。下面这 6 个问题， 每一个都对应着 WorkflowProgram 四层设计中的某一层。

1. 最终要交付哪些 .claude/ 文件？
2. 机器可读真源是什么？（相当于你的 workflow-spec.yaml）
3. 哪些步骤必须沉到确定性脚本？（不能只靠 prompt）
4. 写入边界和冲突处理怎么定义？（怎么做 managed apply）
5. 最终 verdict 由谁给出？（runner 还是独立 judge）
6. 失败经验如何进入下一轮约束？（lessons → constraints）
最小可行版本： 一个机器可读 spec + 一个最小 entry wrapper + 一个边界明确的写入流程 + 一个独立 verdict 产物 + 一个 lessons 文件。这已经比大多数"纯 prompt workflow"稳很多了。

快速上手

5 分钟从安装到第一次运行

1

安装

需要 Python 3.10+ 和 pyyaml。

git clone https://github.com/Logic-70/WorkflowProgram-CN
cd WorkflowProgram-CN
pip install pyyaml
python tools/build_plugin.py
2

启动

在你的目标项目目录中：

cd your-project
claude --plugin-dir /path/to/WorkflowProgram-CN/dist/plugin
3

用自然语言描述需求

workflowprogram-orchestrate 会自动路由到正确的入口：

"为当前项目设计一个 code review workflow"
"审计当前项目的 workflow 结构"
"验证当前项目的 workflow 资产"
4

查看运行结果

运行结束后，在你的项目中查看：

.workflowprogram/runs/<run-id>/
  state.json          # 控制面状态
  events.jsonl        # 事件流
  validation-runtime-report.md

# Design: Target Executor Provider Evidence

## Decision Frame

- Goal: generated target workflows must not depend on `claude -p` as the node executor.
- Scope: target-side runtime generated by WorkflowProgram and shared scripts it delegates to.
- Preserved truth: `target_runtime_policy.mode=managed_runtime` still makes code own graph lifecycle, and `target-runtime-finalizer.py` owns final publish.
- New truth: node execution is selected through `target_executor_policy`.

## Executor Policy

New generated specs declare:

```yaml
target_executor_policy:
  default_provider: current_agent
  allowed_providers:
    - fixture_host
    - command_adapter
    - current_agent
    - manual
  evidence_dir: outputs/stages/executor-evidence
  unsupported_provider_verdict: FAIL
```

`fixture_host` and `command_adapter` are automatic providers. `current_agent` and `manual` are evidence providers: the operator/model performs node work in the current session and writes evidence files under `RUN_ROOT/outputs/stages/executor-evidence/`.

## Manual / Current-Agent Evidence

Each node evidence file must include:

- `schema_version`
- `provider`
- `node_id`
- `status`
- `operator`
- `started_at`
- `completed_at`
- `input_refs`
- `output_refs`
- `outputs[].path`
- optional `outputs[].sha256`

The runner validates evidence before accepting node completion and records artifact provenance. It does not directly mark manual/current-agent runs as PASS. When every node passes evidence checks, the runner returns `BLOCKED` with `manual_finalizer_required=true`.

## Finalizer Promotion

The finalizer may promote `BLOCKED` to `PASS` only when:

- `target-state.json`, `node-results.json`, `artifact-provenance.json`, and lifecycle events are present;
- every manual/current-agent node has valid executor evidence;
- each evidence output has matching provenance;
- required reports pass;
- planned publish sources all come from the current run.

If any doctor/contract/report/provenance check fails, the finalizer writes `target-state.status=FAIL` and does not publish.

## Bypass Semantics

The target command remains wrapper-only. A workflow such as `/stride-audit` may describe usage, but it must invoke `.workflowprogram/runtime/workflow-entry.py`; it must not manually step through nodes or write final reports directly. Directly written reports without current-run state, node results, provenance, and finalizer publish cannot receive clean PASS.

## Review Closure

Round 1:

- Finding: defaulting to `current_agent` could block all runs before finalizer.
- Decision: accepted.
- Fix: runner returns `BLOCKED`, `validate-target-runtime-state.py` allows `BLOCKED` only for manual/current-agent with evidence, and finalizer promotes only after verification.

Round 2:

- Finding: required report `target-runtime-summary.json` would reject `BLOCKED` before finalizer can promote.
- Decision: accepted.
- Fix: templates include `BLOCKED` as an allowed pre-finalizer status for the runner summary.

Round 3:

- Finding: generated wrappers could still carry deprecated `--claude-bin` and accidentally forward it.
- Decision: accepted.
- Fix: keep the CLI flag as a deprecated no-op for compatibility, but never forward or execute it.

Round 4:

- Finding: unsupported provider failure must still leave debuggable runtime evidence.
- Decision: accepted.
- Fix: target runner writes failure `target-state.json`, `target-events.jsonl`, `node-results.json`, `artifact-provenance.json`, and target runtime summary before returning FAIL.

Round 5:

- Finding: command wrappers could bypass runtime by directly writing report directories such as `outputs/stride-audit`.
- Decision: accepted.
- Fix: generated runtime validation treats prompt-heavy direct-report markers as wrapper-only violations.

Latest review found no new actionable design issue.

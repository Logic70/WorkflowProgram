# Design

## Decision Frame

User goal: make workflows generated by WorkflowProgram inherit the same code-controlled execution discipline that WorkflowProgram uses for itself.

Scope boundary: this change covers generated target workflow runtime control. It does not replace WorkflowProgram's product S0..S6 runner and does not migrate any external target repository in this change.

Current truth to preserve:

- Product lifecycle orchestration remains `workflow-entry.py -> workflow-runner.py -> validate-run-state.py`.
- Target workflow shape is already declared in `workflow-spec.yaml.workflow_graph`.
- Generated target runtime assets already live under `.workflowprogram/runtime/`.
- `workflow-spec.yaml` remains the machine-readable runtime map; design prose remains in target design source files.

Output artifacts being changed:

- `workflow-spec.yaml` schema and templates.
- Target runtime generator and validators.
- S5 generated-runtime checks.
- Runtime smoke fixtures and docs.

## Target Runtime Policy

Generated target workflows can declare:

```yaml
target_runtime_policy:
  mode: managed_runtime
  entry_mode: wrapper_only
  graph_source: workflow_graph
  lifecycle_events_required: true
  owner_resolution_failure: terminal
  output_contract_failure: retry_then_terminal
  max_retries_per_node: 3
  artifact_provenance_required: true
  immutable_during_run:
    - .claude/**
    - .workflowprogram/design/**
    - .workflowprogram/runtime/**
    - config/scripts/**
```

The policy means the target workflow's supported entry must start runtime code, and runtime code must execute the target graph. A prompt-heavy command can explain usage, but it must not be the owner of stage lifecycle.

## Target Managed Runtime

The generated target-side runtime remains a shared-control-plane wrapper bundle, but the wrapped shared script changes for target graph execution:

```text
TARGET_ROOT/.workflowprogram/runtime/workflow-entry.py
  -> PLUGIN_ROOT/scripts/target-workflow-runner.py
  -> TARGET_ROOT/.workflowprogram/runtime/validate-run-state.py
  -> PLUGIN_ROOT/scripts/validate-target-runtime-state.py
```

The product `workflow-runner.py` remains responsible for WorkflowProgram S0..S6 only. It is not reused as the target graph executor.

## Node Execution Semantics

V1 executes graph nodes from the selected entrypoint. If `workflow_graph.transitions` is declared, the runner follows the first transition from the current node; terminal targets stop execution. If no transitions exist, it falls back to node list order.

For each node the runner:

- validates non-logical `input_refs` exist before execution;
- resolves `owner` against registry commands, skills, agents, or a `script:<path>` owner;
- runs the owner through the configured provider;
- validates all `output_refs` after execution;
- records lifecycle events and node result;
- records artifact provenance for every generated output;
- retries according to `target_runtime_policy.max_retries_per_node`.

For deterministic providers (`fixture_host`, `command_adapter`) V1 can materialize declared outputs directly for repeatable tests. For `claude_cli`, V1 starts a fresh process with a node-specific prompt and then trusts only file/evidence validation, not the model's verbal claim.

## Bypass And Boundary Semantics

WorkflowProgram cannot globally prevent a model from using arbitrary Claude Code tools outside the supported command. The managed runtime instead makes bypass detectable and non-passing:

- missing `TargetRuntimeStarted` is FAIL;
- missing node lifecycle is FAIL;
- missing provenance for required output is FAIL;
- owner failure followed by later node success is FAIL;
- mutation under `immutable_during_run` is FAIL;
- command file not reduced to a runtime wrapper is FAIL.

## Closure Review

Round 1:

- Finding: Reusing product `workflow-runner.py` for target graphs would mix WorkflowProgram S0..S6 with target workflow nodes.
- Decision: accept.
- Fix: add `target-workflow-runner.py` and keep product runner unchanged.
- Verification: generated target runner wrapper must call `target-workflow-runner.py`, not `workflow-runner.py`.

Round 2:

- Finding: "Only entry" could be misread as global tool sandboxing, which Claude Code plugins cannot fully enforce.
- Decision: accept.
- Fix: define support boundary as wrapper-only official entry plus S5/runtime evidence failure for bypass.
- Verification: validators check lifecycle/provenance evidence instead of relying on prose.

Round 3:

- Finding: Existing target graphs may not declare transitions.
- Decision: accept.
- Fix: V1 follows transitions when present and otherwise uses node list order.
- Verification: fixtures cover a transition-based graph and remain compatible with simple ordered graphs.

Round 4:

- Finding: Immutable path defaults can conflict with historical examples where target graph outputs generated `.claude/**`.
- Decision: accept.
- Fix: templates move target runtime outputs to domain output paths; managed assets remain owned by WorkflowProgram develop, not by target runtime execution.
- Verification: spec fixture updates avoid immutable/output conflicts.

Round 5:

- Finding: Prompt-heavy command files could still be generated even when runtime is managed.
- Decision: accept.
- Fix: `validate-generated-runtime.py` checks the registered main command contains a runtime-entry invocation marker.
- Verification: runtime smoke fixture command is wrapper-only.

Round 6:

- Finding: A runner that only creates files could hide missing owner registration.
- Decision: accept.
- Fix: owner resolution is checked before output materialization; missing owner is terminal.
- Verification: add spec/runtime tests for owner resolution failure.

Round 7:

- Finding: State validation could pass while evidence/provenance is missing.
- Decision: accept.
- Fix: target state validator checks event types, node results, output existence, and provenance.
- Verification: generated state validator wrapper delegates to `validate-target-runtime-state.py`.

Round 8:

- Finding: No new actionable issue found.
- Decision: close.
- Fix: none.
- Verification: design and implementation plan align.
